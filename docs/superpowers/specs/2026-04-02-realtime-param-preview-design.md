# 实时参数预览设计

**日期：** 2026-04-02
**语境：** targetv3 架构

---

## 问题

用户拖动参数滑块时，期望预览面板实时刷新（0.0 → 0.3 过程中每帧都更新），而不是只在松手后刷新。

当前（targetv3 设计中）的 `set_param` 每次调用都入 undo 栈、创建新 Graph 版本。若每帧触发，会产生几十条 undo 记录，不可接受。

---

## 约束

- **AI 节点只能手动执行**：拖动参数不触发 AI 节点，只执行 GPU/Image 节点链。
- **不破坏现有模块边界**：缓存管理器（405）、事件系统（408）、执行器（409/410/411）无需改动。
- **undo 粒度正确**：拖动一次滑块只产生一条 undo 记录（拖动前 → 拖动后）。

---

## 方案：preview/commit 双模式

### 核心思路

引擎 API 的 `set_param` 增加 `preview: bool` 参数：

- **preview=true（拖动中）**：更新参数值 + 触发执行，**不入 undo 栈，不清空 redo 栈**
- **preview=false（松手，commit）**：更新参数值 + 触发执行 + **入 undo 栈，清空 redo 栈**

### 完整数据流

**拖动中（每帧）：**

```
参数控件系统（检测到拖动中的值变化）
  → 画布控制器：PreviewParam { node, key, value }
  → 引擎 API：set_param(node, key, value, preview=true)
  → 402 节点图控制器：修改当前版本参数，跳过 undo 栈，推送 GraphChanged
  → 401 执行调度器：规划器标脏 + 传播，遇 AI 节点跳过
  → 执行运行时：只执行 GPU/Image 节点
  → 405 缓存管理器：写入新结果
  → 事件系统：推送 NodeFinished
  → 交互控制器：通知画布 + 预览面板刷新
  → 预览面板：从缓存读取 Image → 渲染
```

**松手时（一次）：**

```
参数控件系统（检测到拖动结束）
  → 画布控制器：CommitParam { node, key, value }
  → 引擎 API：set_param(node, key, value, preview=false)
  → 402 节点图控制器：当前版本入 undo 栈，创建新版本，清空 redo 栈，推送 GraphChanged
  → （后续执行流程同上）
```

### 各模块改动

| 模块 | 改动 | 说明 |
|------|------|------|
| 引擎 API | `set_param` 加 `preview: bool` | 统一入口，调用方无需关心内部区别 |
| 402 节点图控制器 | preview=true 时跳过入 undo 栈步骤 | 与 `move_node` 的"可选不记录"是同一模式 |
| 401 执行调度器 | 规划时遇 AI 节点标脏但跳过执行 | 延伸现有的 executor 分发机制 |
| 参数控件系统 | 区分"拖动中"和"拖动结束"两种事件 | 控件知道自身的交互状态 |
| 画布控制器 | 转发 PreviewParam / CommitParam 两种消息 | 不做业务判断，透传意图 |

**无需改动：** 缓存管理器（405）、纹理管理器、事件系统（408）、三路执行器（409/410/411）。

### Undo 行为

| 操作 | undo 栈 | 效果 |
|------|---------|------|
| 拖动过程（N 帧 preview） | 不入栈 | 无 undo 记录积累 |
| 松手（commit） | 入栈一次 | 记录拖动前后的完整差异 |
| Cmd+Z | 弹出栈顶 | 直接回到拖动前的参数值 |

### AI 节点边界处理

执行调度器规划时，遇到 AI 节点：

- **标脏**：AI 节点及其下游正常标为脏（参数确实变了）
- **跳过执行**：不放入执行计划，等待用户手动点击运行
- **手动执行时**：完整执行包括 AI 节点

这样 AI 节点上游的 GPU 节点可以实时预览，AI 节点的输出等用户手动触发后再更新。

---

## 设计决策

- **为何选 preview/commit 而非 debounce**：debounce 依赖时间窗口猜测用户意图，不如控件直接报告"拖动中/结束"语义精确。
- **为何 preview 不清空 redo 栈**：拖动是探索性操作，不应破坏用户已有的重做历史；commit 才代表用户确认了新值。
- **为何 AI 节点只能手动执行**：AI 推理耗时秒级，实时触发无意义且浪费资源；用户在调好 GPU 参数后手动触发更符合工作流。
