# 开发工作流程

## 流程总览

```
Issue → 分支 → 设计（复杂任务）→ 开发 → 验证 → 推送 → PR → Code Review → 合并 → 清理
```

## 各步骤规则

### 1. 创建 Issue

每个功能/修复对应一个 GitHub Issue。

### 2. 创建分支 + Worktree

从 main 创建功能分支，使用 worktree 隔离开发。Skill: `superpowers:using-git-worktrees`

命名规范：

| 前缀 | 用途 |
|------|------|
| `feature/描述` | 新功能 |
| `fix/描述` | Bug 修复 |
| `hotfix/描述` | 紧急修复 |
| `refactor/描述` | 代码重构 |
| `docs/描述` | 文档更新 |

### 3. 设计

**触发条件：** 新功能、架构变更、多文件改动。单行修复、文档更新等简单任务跳过。

| 步骤 | Skill |
|------|-------|
| 设计讨论 | `superpowers:brainstorming` |
| 实现计划 | `superpowers:writing-plans` |

设计文档产出到 `docs/superpowers/specs/`，实现计划产出到 `docs/superpowers/plans/`。

### 4. 开发

按实现计划逐步开发。Commit 信息格式：`前缀: 描述 #Issue编号`

| 前缀 | 含义 |
|------|------|
| `feat:` | 新功能 |
| `fix:` | Bug 修复 |
| `docs:` | 文档 |
| `refactor:` | 重构 |
| `test:` | 测试 |
| `chore:` | 杂项 |

每完成一个阶段就提交，不要攒大 commit。

### 5. 验证

推送前必须完成，不可跳过：

1. `cargo test --workspace` — 测试通过
2. `cargo run -p nodeimg-app --release` — 实际运行
3. 手动操作新增/修改的功能
4. 边界情况检查

### 6. 推送 + PR

推送分支到远程，创建 PR。PR 描述中用 `Closes #N` / `Fixes #N` 关联 Issue。

### 7. Code Review

| PR 类型 | 审查方式 |
|---------|---------|
| 简单（bug 修复、文档、小改动） | 快速扫描，检查明显问题 |
| 复杂（新功能、架构变更、多文件改动） | **必须**使用 `superpowers:requesting-code-review` |

有问题 → 修改后 push，PR 自动更新。通过 → 合并。

### 8. 合并 + 清理

合并 PR 到 main。删除远程和本地分支。

## 关键原则

- 每个功能/修复 = 一个独立分支 + 一个 PR
- main 始终是经过审查的干净代码
- 长期分支定期 rebase main

## 易犯错误（经验教训）

以下问题在 #65 开发中暴露，记录以防重复。

### Worktree 文件路径

创建 worktree 后，所有 Write/Edit 操作必须使用 worktree 的绝对路径（`.worktrees/<name>/...`）。**绝不**写到主仓库目录——残留文件会导致合并后 `git pull` 冲突。

### Brainstorming 提问方式

- 每个决策点先用一句话描述**用户可感知的区别**（如"拖滑条时图片是否立刻更新"），再解释技术实现
- 方案对比表（做法、体验、复杂度、推荐）应**前置**，不要先让用户盲选再解释
- 用户说"什么意思"时立刻切换到具体场景描述，不要继续堆术语

### 迁移类任务的依赖枚举

迁移/重构任务在设计阶段必须系统枚举所有依赖点，不能靠直觉：

```bash
# 示例：找出所有使用某个类型的位置
grep -rn "EvalEngine\|AiExecutor\|Connection" crates/nodeimg-app/
```

列成清单逐个确认是否需要迁移。漏掉的依赖会在 spec review 阶段才暴露，浪费来回。

### 合并后清理顺序

1. `gh pr merge` — 合并 PR
2. `git worktree remove .worktrees/<name>` — 先删 worktree
3. `git branch -d <branch>` — 再删本地分支（worktree 存在时无法删除分支）
4. `git pull` — 同步 main
