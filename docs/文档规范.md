# 文档书写规范

本项目所有设计文档遵循以下规范。

---

## 核心原则

- **声明式** — 描述"是什么"和"规则是什么"，不描述"怎么实现"（实现细节属于代码）
- **结构化** — 用表格、列表、树形图承载信息，避免长段叙述
- **无废话** — 每句话必须传递信息，删掉不影响理解的句子就应该删掉

---

## 文档类型与结构

### 领域文档（domain.md）

定义核心概念和规则。

**结构：**
- 按概念分章，每章独立
- 概念按依赖顺序排列（先定义的被后面引用，不循环）
- 每个概念包含：定义 → 分类/属性（表格） → 规则（短句列表）→ 扩展性说明

### 架构文档（architecture.md）

定义组件划分和接口约定。

**结构：**
- 开头：组件总览表（组件名 + 一句话职责）
- 每个组件三段式：
  1. **接口约定** — 这个组件对外提供什么操作
  2. **依赖** — 它依赖哪些其他组件
  3. **内置注册** — 它默认包含什么内容
- 结尾：依赖关系图（ASCII）+ 文件结构（树形图）

### 目录文档（catalog.md）

列举所有同类实体的完整规格。

**结构：**
- 按分类分组
- 每个实体使用**完全相同的表格格式**
- 零叙述，纯声明——可直接作为 API reference 查阅

### 子系统文档（gpu.md 等）

描述独立子系统的架构和规范。

**结构：**
- 技术栈（用什么）
- 组件结构（表格）
- 执行流程（编号步骤或流程图）
- 规范和约定（代码模式、命名规则）
- 状态表（哪些已实现、哪些未实现）

---

## 格式约定

| 信息类型 | 使用格式 |
|---------|---------|
| 多个实体的属性对比 | 表格 |
| 有序步骤 | 编号列表 |
| 规则/要点 | 无序列表，每条一个要点 |
| 代码结构 | 树形图（缩进 + 竖线） |
| 依赖/流程关系 | ASCII 图 |
| 代码示例 | 围栏代码块，标注语言 |

**不使用：**
- 长段落叙述（超过 3 行的纯文字段落应拆为列表或表格）
- 主观评价（"这个设计非常优雅"）
- 实现细节（具体算法步骤、代码行号引用）

---

## 标注约定

- 未实现的功能在描述后标注 **（未实现）** 或 **（v2）**
- 文档先行于代码时，相关 Issue 编号标注在章节中
- 跨文档引用使用 `详见 xxx.md`

---

## 检查清单

写完一篇文档后，对照以下项目检查：

- [ ] 每个章节是否可以独立阅读
- [ ] 表格是否比当前的文字段落更适合表达这段信息
- [ ] 是否有句子删掉后不影响理解（如果有，删掉）
- [ ] 概念是否按依赖顺序排列（不前向引用未定义的概念）
- [ ] 同类实体是否使用了完全相同的格式
