# 基本概念

> GUI 的分层基本概念及其精确定义。所有上层设计都建立在这些概念之上。

## 层级总览

```
图元 (Primitive)                 纯渲染单元，可带 id 和 gestures，直接在 Desc 中使用
                                  Text, Image, Icon, Circle, Line, Curve, Path,
                                  Grid, Connection, CustomPaint
  ↑
原子控件 (Atomic Widget)          最小功能单元（Brad Frost），直接组合 Container + Leaf
                                  Button, Slider, TextInput, Toggle, Dropdown, ImageViewer
  ↑
组合控件 (Composite Widget)       多个原子控件聚合的更大功能单元
                                  ColorPicker, NumberInput, SearchBox, Group, PopupPanel
  ↑
特定框架控件 (Framework Widget)   应用顶层结构单元，带内容槽
                                  Panel, NodeCard
```

除上述四级概念外，还有一个贯穿所有层级的基础结构：**容器 (Container)**。容器不是控件（无交互、无 Props），而是所有上层概念的结构骨架。

## 判定规则

判定基于 `NodeKind` 和**设计意图**，不是基于子节点数量或运行时查询。

```
NodeKind?
├── Leaf(LeafKind)  → 图元
├── Container       → 容器
└── Widget(props)   →
    ├── 功能单一（一个东西）      → 原子控件
    └── 多个功能单元聚合          →
        ├── 应用顶层结构单元      → 特定框架控件
        └── 否                    → 组合控件
```

原子、组合、特定框架这三级的区分是**语义/意图**的，不是结构的。同一段 `build()` 代码，分类取决于设计意图，而不是展开后的子树形状。

## 图元 (Primitive)

树的叶子节点。不可有子节点。无交互、无状态、不产出 Action。由特定渲染管线绘制。

判据：`NodeKind::Leaf(LeafKind)`

### 基础文本/图像

| 图元 | Props | 渲染管线 | 命中测试 |
|------|-------|---------|---------|
| Text | content, font_size, color, weight, alignment | text | 不可命中 |
| Image | texture, fit, tint | image | 矩形 |
| Icon | icon_id, size, color | svg / text（图标字体） | 矩形 |

### 几何图形

Container + Decoration 搞不定的非盒模型形状。

| 图元 | Props | 渲染管线 | 命中测试 |
|------|-------|---------|---------|
| Circle | radius, fill, stroke | circle | 圆形距离 |
| Line | start, end, width, color, dash | curve | 线段距离 |
| Curve | points, stroke, dash | curve | 曲线距离 |
| Path | commands, fill, stroke | curve | 路径包含 |

### 应用特化

| 图元 | Props | 渲染管线 | 命中测试 |
|------|-------|---------|---------|
| Grid | spacing, dot_color, dot_size | circle（批量） | 不可命中 |
| Connection | from_port: NodeId, to_port: NodeId | curve | 曲线距离 |

Connection 是 Curve 的特例——只存两个端口引用，绘制和命中测试时自动查找端口位置算贝塞尔曲线。

### 逃生舱口

| 图元 | Props | 用途 |
|------|-------|------|
| CustomPaint | paint_fn: Box<dyn CustomPainter> | 自定义绘制（直方图、色盘、波形图等） |

任何需要特殊绘制但不值得专门建图元的情况走这条路。对标 Flutter 的 `CustomPainter`、SwiftUI 的 `Canvas`。

### 盒模型 vs 非盒模型的选择

一个形状是用 **Container + Decoration** 还是做成 **Leaf 图元**？判据：

- **盒模型形状**（矩形/圆角矩形，可通过 padding/border/radius/shadow 表达）→ Container + Decoration
- **非盒模型形状**（圆、线、任意路径、自定义绘制）→ Leaf 图元

下面这些都是盒模型，不占用 LeafKind：

| 形式 | 实现方式 |
|------|---------|
| 矩形 | Container 本身 |
| 圆角矩形 | `decoration.radius` |
| 边框 | `decoration.border` |
| 阴影 | `decoration.shadow` |
| 渐变背景 | `decoration.background` 扩展支持 LinearGradient/RadialGradient |
| 椭圆 | Circle 的宽高不等特例，或用 Path |
| 分割线 | 带底边框的定高 Container |
| 滚动区 | `overflow: Scroll` 的 Container |

图元可以直接作为 `Desc::Leaf` 在控件的 `build()` 内部使用，也可以作为独立的 Desc 节点。Leaf 节点本身支持：

- **id** —— 参与 reconcile，跨帧保持身份
- **gestures** —— 响应命中测试和手势识别（通过 BoxStyle 上的 gestures 字段）

这意味着一段需要点击的文字直接写 `Desc::Leaf { kind: Text, gestures: vec![Tap] }` 即可，不需要额外的包装层。Button 内部的标签文字、节点端口的圆点、画布的背景网格、节点间的连线——这些都是原子控件的 `build()` 直接产出的 `Desc::Leaf`。

## 容器 (Container)

树的内部节点。有子节点，负责布局和视觉装饰。无交互逻辑，不产出 Action。是所有上层概念的结构骨架。

判据：`NodeKind::Container`

> 注：BoxStyle 和 gestures 等属性在数据结构层面是所有节点共通的，但主要使用者是容器（因为容器有子节点需要布局，有装饰需要渲染）。Leaf 节点通常只用其中的 width/height 部分。

### 布局属性 (BoxStyle)

| 属性 | 说明 |
|------|------|
| position | Flow（参与 Flexbox）/ Absolute { x, y }（自由定位） |
| width, height | Fixed(f32) / Auto / Fill |
| min/max width/height | 尺寸约束 |
| padding, margin | 四边内外间距 |
| direction | Row / Column（Flexbox 主轴） |
| gap | 子节点间距 |
| align_items | 交叉轴对齐 |
| justify_content | 主轴分布 |
| flex_grow | 剩余空间分配权重 |
| overflow | Visible / Hidden / Scroll |
| transform | 坐标变换（用于相机），影响所有后代的绘制和命中测试 |

### 视觉属性 (Decoration)

| 属性 | 说明 |
|------|------|
| background | 背景色 / 渐变 |
| border | 边框（宽度、颜色） |
| radius | 四角圆角 |
| shadow | 阴影（颜色、偏移、模糊、扩展） |

### 交互属性

| 属性 | 说明 |
|------|------|
| hittable | 是否参与命中测试（默认：有 Decoration 则可命中） |
| gestures | 命中后注册哪些手势识别器（Tap、Drag、LongPress、Resize） |

**这些属性对所有节点（Container 和 Leaf）都可用**。一个 Leaf 节点也可以声明 `hittable: true` 和 `gestures: vec![Tap]`，从而获得交互能力。这让简单的"可点击文字"或"可拖拽图片"不必套一层 widget。

容器不是控件。Separator（带底边框的定高容器）、Spacing（定尺无装饰容器）、ScrollView（Overflow::Scroll 容器）都只是容器的样式配置，不需要独立概念。

## 控件 (Widget)

有 Props 和 `build()` 的交互单元。`build()` 将 Props 展开为子树。响应手势，产出 Action。

判据：`NodeKind::Widget(Box<dyn WidgetProps>)`

```rust
trait WidgetProps {
    fn build(&self, id: &str) -> Desc;    // 展开为子树
    fn props_eq(&self, other: &dyn WidgetProps) -> bool;  // 变更检测
}
```

控件按**语义角色**分为三级：原子控件 → 组合控件 → 特定框架控件。采用 Brad Frost 的 Atomic Design 方法论——从"用户眼中这是一个什么东西"来划分，而不是按实现结构。简单的显示需求（纯文字、图标、图片）直接用图元 `Desc::Leaf`，不需要封装成 widget。

### 原子控件 (Atomic Widget)

最小的**功能单元**，从用户视角看是"一个东西"。主要目的是**提供一种交互**。`build()` 直接组合 Container 和 Leaf 子树，这是最常见的实现模式。

采用 Brad Frost 的 Atomic Design 定义——Button 永远是原子控件，不管内部怎么实现。分类基于功能语义，不基于 build() 展开的具体结构。

| 原子控件 | Props | 产出 Action |
|---------|-------|------------|
| Button | label, icon, disabled | Click |
| Slider | label, range, step, value, disabled | Change(f32) |
| TextInput | label, value, disabled | Change(String) |
| Toggle | label, value, disabled | Toggle(bool) |
| Dropdown | options, selected, disabled | Select(usize) |
| Checkbox | label, checked, disabled | Toggle(bool) |
| ImageViewer | image | Zoom(f32), Pan(x, y) |

### 组合控件 (Composite Widget)

多个原子控件聚合而成的更大功能单元。从用户视角看是"多个东西协作完成一件事"。

| 组合控件 | 组成 | 产出 Action |
|---------|------|------------|
| ColorPicker | Slider x3 + TextInput + 色盘 CustomPaint | Change(Color) |
| NumberInput | Label + TextInput + 拖拽调节 | Change(f32) |
| SearchBox | TextInput + 搜索按钮 + 下拉结果列表 | Query(String) |
| TitleBar | Label + Spacing(Fill) + Button("close") | Click |
| Group | Label + Toggle + 内容槽 | Toggle(bool) |
| PopupPanel | 锚定定位 + 内容槽 | — |

组合控件可以包含内容槽（如 Group、PopupPanel 接受 `Vec<Desc>` 作为内容），也可以是固定结构（如 ColorPicker 内部完全固定）。内容槽只是 Props 的一种设计选择，不改变组合控件的本质。

### 特定框架控件 (Framework Widget)

**应用顶层结构单元**，带内容槽。和组合控件的结构机制相同（都有 Props + content slot），但承担特殊的架构角色：它们是应用顶层的结构骨架，而不是可复用的 UI 构件。

| | 组合控件 | 特定框架控件 |
|---|---------|-------------|
| 角色 | 可复用 UI 构件 | 应用顶层结构 |
| 实例数 | 任意多 | 通常是单例或每实例独立 |
| 持有者 | 父级控件 | PanelRoot / Canvas Nodes 容器 |
| 内容来源 | 父级传参或写死 | 应用状态 / NodeDef 自动映射 |
| 代码组织 | `widget/composites/` | `panel/`, `canvas/` |

本框架有两个特定框架控件：

**Panel（面板）** — 通常放置在 PanelRoot 下的顶层结构。提供标题栏、阴影边框、resize 边缘、拖拽能力。

```
panel("属性", content) → Container(Absolute, shadow+border)
                           ├── TitleBar(draggable) — Label + CloseButton
                           └── Content(Scroll) — ...注入的内容...
```

典型实例：Toolbar、Properties、Preview。

**NodeCard（节点卡片）** — 通常放置在 Canvas 下的顶层结构。提供卡片装饰、标题栏、端口列表。内容从 NodeDef 自动映射。

```
node_card(node_def) → Container(Absolute, card decoration)
                        ├── TitleBar — Label(node_def.name)
                        ├── Ports — port entries from node_def
                        ├── Content — widgets mapped from node_def.params
                        └── Preview — optional thumbnail
```

Panel 和 NodeCard 的结构机制完全一致，唯一差异是**常驻的放置位置**。坐标系（屏幕坐标 vs 世界坐标）由祖先节点的 Transform 决定——理论上把一个 Panel 放到 CanvasRoot 下就会变成世界坐标的浮动窗，这种灵活性刻意保留。

所有能力（自由定位、拖拽、缩放、阴影、圆角）都来自容器的通用属性和手势系统，没有框架控件专属的实现。

## 无限嵌套

控件系统天然支持无限嵌套，和 Figma 的 Frame 嵌套模型一致。嵌套在两个层面都成立：

### 结构层面（树的能力）

控件的 `build()` 返回 `Desc`，`Desc` 可以包含任何 `Desc::Widget` 节点。reconcile 递归展开每一层 Widget 节点，没有深度限制。

```rust
impl WidgetProps for PanelProps {
    fn build(&self, id: &str) -> Desc {
        Desc::Container {
            children: vec![
                Desc::Widget { props: Box::new(TitleBar { ... }) },  // 嵌套组合控件
                Desc::Widget { props: Box::new(Panel { ... }) },     // 嵌套 Panel
                // ... 任意深度
            ],
            ..
        }
    }
}
```

### 内容槽层面（带内容槽的控件）

```rust
panel("外层", vec![
    panel("中层", vec![
        panel("内层", vec![
            color_picker(...),
            slider(...),
        ]),
    ]),
])
```

每一层都是独立的结构单元，可以无限包裹。

### 与 Figma 的对比

| | Figma | 本框架 |
|---|-------|-------|
| 结构嵌套 | Frame → Frame → ... | Widget → Widget → ... |
| 自动布局 | Auto Layout | Flexbox |
| 自由定位 | Frame 内的子元素 | Absolute position |
| 组件复用 | Component + Instance | 函数调用（写一次，调多次） |
| 变体/状态 | Variants | InteractionState + Props |
| 覆写 | Instance overrides | Props 参数 |

结构能力完全对等。缺的是 Figma 的工具链（组件库 UI、可视化覆写、变体切换）——这些是 IDE 级功能，不是框架能力。

### 嵌套约束

- **ID 唯一性** — 每个 Widget 需要在树中有唯一 ID。嵌套时调用者要给注入的内容起不重复的 ID，通常用路径前缀（`"panel_1/section_a/slider"`）
- **reconcile 按 ID 匹配** — 嵌套层级变了但 ID 不变，组件会被当作"同一个实例"复用；ID 变了就是销毁+创建
- **性能** — 理论上深度无限，实际几百层递归才会有栈溢出风险，正常 UI 不会触碰

## 树中的映射

从树的角度只有三种节点类型：

```rust
enum NodeKind {
    Container,                     // 容器
    Leaf(LeafKind),                // 图元
    Widget(Box<dyn WidgetProps>),  // 控件（原子/组合/特定框架）
}
```

原子控件、组合控件、特定框架控件在树里**都是 Widget 节点**，区别在于 Props 实现和 `build()` 展开的子树复杂度、以及承担的语义角色。树不关心你是原子还是组合——它只做 diff、layout、paint、hit test。

控件的分级是**给人看的**（用于代码组织、沟通、设计决策），不是给框架看的。框架对所有 Widget 一视同仁。
