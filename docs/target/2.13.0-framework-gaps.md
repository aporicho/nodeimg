# 框架缺口分析：已有 vs 待建

> 对照 widget-anatomy 的八个组成部分和框架级能力，盘点当前代码实际状态。

## 已有能力

| 能力 | 实现位置 | 状态 |
|------|---------|------|
| 身份（id + props_hash） | `panel/tree/desc.rs` | 完整 |
| 配置（Desc 字段） | `panel/tree/desc.rs` | 完整 |
| 布局（Flexbox） | `widget/layout/` + `LayoutTree` trait | 完整 |
| 绘制（Container decoration + Leaf 图元） | `panel/tree/paint.rs` | 完整 |
| 命中测试（decoration 命中） | `panel/tree/hit.rs` | 完整 |
| 生命周期（reconcile 驱动） | `panel/tree/diff.rs` | 完整 |

| 无限画布（相机/平移/网格背景） | `canvas/` | 完整 |
| 按钮控件函数 | `demo.rs` 中的 `button()` | 仅 demo |

## 缺口

### 一、交互状态机

**现状：** 控件没有 hover/pressed/focused 状态。鼠标悬停按钮无视觉反馈。

**需要：**

```
         鼠标进入
Normal ──────────→ Hovered
  ↑                  │ 鼠标按下
  │ 鼠标离开          ↓
  ←──────────── Pressed
                     │ 鼠标松开
                     ↓
              触发 Action + 回到 Hovered
```

每个可交互节点需要存储一个 `InteractionState` 枚举。框架统一管理状态转换（不是控件自己管）。

**涉及：**
- PanelNode 加 `interaction_state` 字段
- 框架在 event 阶段根据鼠标位置 + 事件类型自动转换状态
- paint 根据状态选择不同的视觉样式（如 hover 时背景变亮）
- 需要捕获机制：按下时锁定事件目标，直到松开

### 二、事件回调

**现状：** hit_test 返回节点 id，demo.rs 手动 match id 做处理。每加一个按钮就加一段 if/else。

**需要：** 通用的 Action 产出 + 路由机制。

```
鼠标松开
  → 框架检测：当前 Pressed 的节点是谁？
    → 产出 Action::Click("btn_a")
      → 路由给面板的 route() 函数处理
```

控件函数在构造 Desc 时声明"我能产出什么 Action"，框架负责在正确的时机触发。

**涉及：**
- `Action` 枚举：Click、Change(f32)、Change(String)、Toggle(bool)、Select(usize)
- Desc 节点可以携带事件声明（哪些 Action 能从这个节点产出）
- 面板实例定义 route 函数处理 Action
- 当前 demo.rs 的 match 逻辑迁移到 route 模式

### 三、滚动

**现状：** `scroll_offset` 和 `content_height` 存在 PanelNode 上，layout 的 arrange 会读写它们。但没有代码把滚轮事件写入 `scroll_offset`。`overflow: Scroll` 的容器也没有 clip。

**需要：**
- Shell 翻译滚轮事件为 `AppEvent::Scroll { x, y, delta }`
- 框架在 event 阶段找到鼠标所在的滚动容器，调 `scroll()` 更新 offset
- paint 对 `overflow: Scroll` 的容器做 `push_clip` / `pop_clip`

**涉及：**
- `shell/translator.rs` 加滚轮事件翻译
- `shell/event.rs` 加 `Scroll` 事件类型
- event 处理中加滚动容器查找逻辑
- `panel/tree/paint.rs` 加 clip 支持

### 四、更多控件函数

**现状：** 只有 `button()`。

**需要（按优先级）：**

| 控件 | 树结构 | 需要的新能力 |
|------|--------|------------|
| text_display | Text 叶子 | 无，最简单 |
| separator | Container(decoration: 细线) | 无 |
| toggle | Container(Row) + 开关轨道/圆点 | 交互状态 + Click action |
| slider | Container(Row) + 轨道 + 滑块 + 值文字 | 交互状态 + 拖拽 + Change action |
| text_input | Container(decoration) + Text | 交互状态 + 焦点 + 文字编辑 + Change action |
| dropdown | Container + 弹出列表 | 交互状态 + 焦点 + 弹出层 |
| group | Container(Column) + 折叠展开 | Toggle action + 业务状态 |
| list_view | Container(Scroll) + 子项 | 滚动 + Select action |
| image_viewer | Image 叶子 | Image LeafKind |

依赖关系：
```
text_display, separator → 无前置依赖
toggle → 交互状态 + 事件回调
slider → 交互状态 + 事件回调 + 拖拽捕获
text_input → 交互状态 + 事件回调 + 焦点 + 文字编辑
dropdown → text_input 的全部 + 弹出层
```

### 五、Image 叶子

**现状：** `LeafKind` 只有 `Text`。预览面板和节点缩略图需要 Image。

**需要：**
```rust
LeafKind::Image {
    texture: Arc<wgpu::TextureView>,
    // 尺寸由 BoxStyle 的 width/height 决定
}
```

**涉及：**
- `widget/layout/types.rs` 加 `LeafKind::Image`
- `panel/tree/paint.rs` 加 `draw_image` 分支
- `renderer` 的 ImagePipeline 已经存在，只需接入

### 六、焦点系统

**现状：** `widget/focus.rs` 有 `FocusState` 结构但未接入树。

**需要：**
- 全局焦点跟踪：哪个节点当前持有焦点
- Tab 键切换焦点到下一个可聚焦节点
- Escape 清除焦点
- 有焦点的节点接收键盘事件

**涉及：**
- 焦点状态存在哪？可以存在面板级别（一个面板有一个焦点节点 id）
- PanelNode 需要标记"是否可聚焦"
- event 阶段：键盘事件分发给当前焦点节点

### 七、文字编辑

**现状：** `widget/text_edit.rs` 有 `TextEditState`（光标位置、选区、编辑内容）但未接入树。

**需要：**
- TextInput 控件获得焦点后进入编辑模式
- 键盘输入插入/删除字符
- 光标移动、选区、复制粘贴
- 编辑结果产出 `Action::Change(String)`

**依赖：** 焦点系统必须先完成。

### 八、主题系统

**现状：** 颜色硬编码在 `demo.rs` 的常量里（BORDER_COLOR、TEXT_COLOR 等）。

**需要：**
- 主题 token 集中管理（背景色、边框色、hover 色、字号等）
- 控件函数读 token 而非硬编码
- 换肤 = 替换一套 token

**优先级不高，但越早做越好** — 每新增一个控件就多一处硬编码颜色，后期统一改造成本高。

## 建议实施顺序

```
第一步：交互状态机 + 事件回调
        ↓（解锁交互能力）
第二步：主题系统
        ↓（后续控件都用 token）
第三步：滚动 + clip
        ↓（解锁长列表）
第四步：控件扩展（toggle → slider → text_display → separator）
        ↓
第五步：焦点 + 文字编辑 + text_input
        ↓
第六步：Image 叶子
        ↓
第七步：dropdown、group、list_view
```

第一步是关键瓶颈——没有交互状态和事件回调，其他控件都做不了。
